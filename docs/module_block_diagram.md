# Module: Block Diagram Builder

## What It Does

The Block Diagram Builder lets engineers create high-level system architecture diagrams — the kind drawn on a whiteboard at project kickoff — and persist them as structured data. Diagrams can be generated by AI from a free-text description, from a list of parts in the library, or from an uploaded architecture PDF. Saved diagrams can be exported as Xpedition netlist seed scripts or connection CSVs.

## Data Model (`models/block_diagram.py`)

```python
class Port(BaseModel):
    name: str
    direction: str  # "input" | "output" | "bidirectional"
    signal_type: str | None  # e.g. "LVDS", "SPI", "power"

class Block(BaseModel):
    id: str          # UUID
    name: str        # e.g. "FPGA", "DDR4 SDRAM"
    category: str    # e.g. "processor", "memory", "power", "connector"
    part_number: str | None  # links to library if known
    ports: List[Port]
    position: dict   # {x: float, y: float} for canvas layout

class Connection(BaseModel):
    id: str
    from_block: str   # Block.id
    from_port: str    # Port.name
    to_block: str
    to_port: str
    net_name: str | None
    signal_type: str | None

class BlockDiagram(BaseModel):
    id: str | None           # None until saved
    name: str
    blocks: List[Block]
    connections: List[Connection]
    created_at: str | None
    updated_at: str | None
```

## API Endpoints

### CRUD

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/diagrams` | List all (summary: id, name, block_count, connection_count, timestamps) |
| `POST` | `/api/diagrams` | Create with full BlockDiagram body |
| `GET` | `/api/diagrams/{id}` | Fetch one by UUID |
| `PUT` | `/api/diagrams/{id}` | Full replace (save after editing) |
| `DELETE` | `/api/diagrams/{id}` | Delete |

### AI Generation

#### `POST /api/diagrams/generate`
**Body:** `{part_numbers: [str], diagram_name: str}`

Fetches matching parts from the library, builds an AI prompt describing each part (name, category, package, summary), and asks the LLM to design a block diagram showing how they interconnect.

**How the LLM is guided:**
- System prompt shows the exact `BlockDiagram` JSON schema
- Instructs the model to infer connections from part types (e.g. FPGA connects to DDR4 via a memory bus, to power IC via power rail)
- Instructs assignment of `category` values: processor, memory, power, comm, sensor, connector, passive
- Instructs port naming conventions: `VIN`, `GND`, `SDA`, `SCL`, `D+`, `D-`, etc.

The generated diagram is auto-saved and its full data returned.

#### `POST /api/diagrams/generate-from-text`
**Body:** `{description: str, diagram_name: str}`

Free-text description of the system. The LLM generates the full `BlockDiagram` JSON from the description. Example input: "A radiation-hardened FPGA controlling two DDR4 banks, a SpaceWire interface ASIC, and two LDO regulators."

#### `POST /api/diagrams/generate-from-pdf`
**File:** PDF architecture document

Extracts text via PyMuPDF, then delegates to `generate_from_text()`. Works on architecture spec documents, interface control documents (ICDs), or conceptual design reports.

### Export

#### `POST /api/diagrams/{id}/export-netlist?format=script`
Generates a Python script for Xpedition that:
- Creates schematic symbols for each block
- Places them on the schematic sheet
- Draws nets between connected ports

#### `POST /api/diagrams/{id}/export-netlist?format=csv`
Generates a connection CSV (netlist seed):
```
From_Block,From_Port,To_Block,To_Port,Net_Name,Signal_Type
FPGA,SDA,EEPROM,SDA,I2C_SDA,I2C
FPGA,SCL,EEPROM,SCL,I2C_SCL,I2C
```

## `services/block_diagram_store.py`

Stores all diagrams in `backend/data/diagrams.json` as a JSON array. Each entry is a serialized `BlockDiagram` dict with an added UUID `id`, `created_at`, and `updated_at` timestamp.

Functions:
- `list_all()` — load and return full array
- `get_by_id(id)` — linear scan by UUID string
- `create(data)` — generate UUID + timestamps, append, save
- `update(id, data)` — find by ID, replace, update `updated_at`, save
- `delete(id)` — find by ID, remove, save; returns `True` if found

All writes are protected by `threading.Lock()`. The store follows the same pattern as `part_library.py`.

## `services/block_diagram_generator.py`

### `generate_from_parts(parts, diagram_name) → BlockDiagram`

Builds a prompt by summarizing each part:
```
Part: TPS7H1111-SEP
Manufacturer: Texas Instruments
Summary: Low-dropout regulator for space applications
Package: SOT-23-5
Category inferred: power regulator
```

The LLM returns a JSON `BlockDiagram` which is validated with `BlockDiagram.model_validate()`. If validation fails, a salvage attempt extracts whatever structure is present.

### `generate_from_text(description, diagram_name) → BlockDiagram`

Directly sends the description. Includes few-shot examples in the system prompt showing a simple 3-block diagram to anchor the response format.

## `services/block_diagram_export.py`

### `generate_netlist_csv(diagram_dict) → str`

Iterates `diagram["connections"]`, formats each as a CSV row. Header: `From_Block,From_Port,To_Block,To_Port,Net_Name,Signal_Type`.

### `generate_netlist_script(diagram_dict) → str`

Generates Python using Xpedition's COM API:
- `mgc.Application` dispatch
- `ActiveDesign.Schematic` access
- Symbol placement (`PlaceSymbol`)
- Net drawing (`DrawNet`)

The script is a scaffold — engineers typically need to adjust symbol names to match their Xpedition library, but the connectivity is correct.

## Frontend — `pages/BlockDiagram.jsx`

**Modes:**
1. **Browse saved diagrams** — list of saved diagrams with block/connection counts
2. **Generate from library** — multi-select checkboxes from library parts → POST to `/api/diagrams/generate`
3. **Generate from text** — free-text input → POST to `/api/diagrams/generate-from-text`
4. **Generate from PDF** — upload zone → POST to `/api/diagrams/generate-from-pdf`

**Diagram viewer:**
- After generation, shows a visual block diagram rendered as an SVG/canvas
- Blocks are positioned using the `position: {x, y}` data from the model
- Connections drawn as lines between block ports
- Block category determines the icon/color

**Export controls:**
- "Export Script" → downloads `.py` file
- "Export CSV" → downloads `.csv` netlist seed
